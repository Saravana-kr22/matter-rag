## 1. Overview

The Access Control Cluster exposes a data model view of a Node's Access Control List (ACL), which codifies the rules used to manage and enforce Access Control for the Node's endpoints and their associated cluster instances. Access to this Access Control Cluster itself requires a special Administer privilege level, such that only Nodes granted such privilege (hereafter termed "Administrators") can manage the Access Control Cluster.

**Cluster ID:** `0x001F` (AccessControl)
**PICS Code:** `ACL`
**Classification:** Base | Utility | Node scope

The cluster manages three primary constructs:
- **ACL (Access Control List):** Entries granting privilege to subjects for targets on the Node.
- **ARL (Access Restriction List):** Restrictions imposed by a managed device (ManagedDevice feature) that forbid specific fabric clients from accessing certain data model elements.
- **AuxiliaryACL:** Auto-generated ACL entries synthesized by other features (e.g., Groupcast cluster), exposed for transparency.

## 2. Protocol Details

### Placement
The Access Control Cluster SHALL be present on the root node endpoint of each Node, and SHALL NOT be present on any other Endpoint of any Node.

### Features
| Bit | Code | Feature | Conformance | Summary |
|-----|------|---------|-------------|---------|
| 0 | EXTS | Extension | O | Device provides ACL Extension attribute |
| 1 | MNGD | ManagedDevice | desc | Device is managed |
| 2 | AUX | Auxiliary | P, desc | Some auxiliary ACL entries may be auto-generated by other features |

### Privilege Hierarchy
Privileges are hierarchical — each level implicitly grants all lower levels:
- **View (1):** Can read and observe all (except Access Control Cluster)
- **Proxy View (2):** [deprecated]
- **Operate (3):** View privileges, and can perform the primary function of this Node (except Access Control Cluster); implicitly grants View
- **Manage (4):** Operate privileges, and can modify persistent configuration of this Node (except Access Control Cluster); implicitly grants Operate & View
- **Administer (5):** Manage privileges, and can observe and modify the Access Control Cluster; implicitly grants Manage, Operate & View

### Access Control Privilege Granting
Each Access Control Entry codifies a single grant of privilege on this Node, and is used by the Access Control Privilege Granting algorithm to determine if a subject has privilege to interact with targets on the Node.

During commissioning, when no ACL entries exist yet, the Access Control Privilege Granting algorithm behaves as if an implicit PASE-based Administer entry is present on the Commissionee (but not on the Commissioner).

### ManagedDevice Feature Flow
Devices with the ManagedDevice feature impose access restrictions on each new fabric via an Access Restriction List (ARL). The `ReviewFabricRestrictions` command triggers an external review flow:
1. Fabric administrator sends `ReviewFabricRestrictions` command.
2. Server sends `ReviewFabricRestrictionsResponse` containing a correlation token.
3. Within 1 hour, server generates `FabricRestrictionReviewUpdate` event (may include `Instruction` text and `ARLRequestFlowUrl`).
4. User may need to interact with an external service or UI to approve/deny restriction changes.
5. ARL attribute is updated to reflect the outcome; a response is returned via the `MTaer` field of the `callbackUrl`.

### Administration Guidelines
The cluster is passive — it only maintains entries. Administrators are responsible for creating and maintaining policy. The cluster SHALL NOT change or remove Access Control Entries of its own volition.

### Default Values
The default values of the ACL and Extension attributes are empty lists (zero elements).

## 5. Security Considerations

### Access Requirements
The Access Control Cluster itself requires the Administer privilege to read or modify. The Administer privilege SHALL NOT be used on Access Control Entries which use the Group auth mode.

### ManagedDevice and Fabric Vendor Verification
A malicious fabric administrator could attempt to create a fabric with a VID that it does not own in order to gain enhanced access. For this reason the device SHOULD perform the Fabric Table Vendor ID Verification Procedure.

A well-implemented web service or app SHOULD validate that the VID in flow URL invocations match recent requests for review by comparing with VID and token pairs from recent ReviewFabricRestrictions requests.

An Administrator supporting the ARL flow should realize that if the device serving this cluster is malicious, it could send dangerous URLs to the client which could take the user to malicious sites. The Administrator SHOULD provide details, such as the VID, to the user about the device sending the event so that the user can make an informed decision about whether to trust the URL.

### ARLRequestFlowUrl Transport Security
The syntax of the ARLRequestFlowUrl field SHALL follow the syntax as specified in RFC 1738 and SHALL use the https scheme for internet-hosted URLs.

An invalid URL example from the spec: `http://company.domain.example/matter/arl/vFFF1p1234` — http scheme is not allowed.

### CASE Authenticated Tag (CAT) Usage
If a set of Nodes is commonly used in entries, then an Administrator MAY consider using CASE Authenticated Tags (CATs) for those entries, especially if membership in the set of Nodes changes over time.

### DataVersion Conditional Writes
Administrators SHOULD take care to use DataVersion conditional writes when administering the list or its contents.

### Commissioning Implicit Entry
During commissioning, an implicit Access Control Entry is present on the Commissionee (but not on the Commissioner) that grants Administer access via PASE, enabling the initial ACL entry to be added.

## 6. Error Handling

### CONSTRAINT_ERROR
The Access Control Cluster SHALL fail to write, and return an appropriate error, if an attempt is made to create or update an Access Control Entry or Access Control Extension such that it would have invalid contents. Conditions that produce CONSTRAINT_ERROR for Access Control Entries:

- Privilege enum value invalid
- AuthMode enum value invalid
- AuthMode is PASE (reserved for future use)
- Subjects element invalid (e.g., illegal CAT with CASE AuthMode)
- Targets element invalid (e.g., no field present; both Endpoint and DeviceType fields present)
- Field combinations invalid (e.g., Administer Privilege with Group AuthMode)

Conditions that produce CONSTRAINT_ERROR for Access Control Extensions:
- There is an attempt to add more than 1 entry associated with the given accessing fabric index in the extension list
- Data value exceeds max length
- Data value not valid TLV-encoded data

### RESOURCE_EXHAUSTED
- An attempt to add an Access Control Entry when no more entries are available SHALL result in a RESOURCE_EXHAUSTED error being reported and the ACL attribute SHALL NOT have the entry added to it.
- An attempt to create an entry with more subjects than the node can support SHALL result in a RESOURCE_EXHAUSTED error and the entry SHALL NOT be created.
- An attempt to create an entry with more targets than the node can support SHALL result in a RESOURCE_EXHAUSTED error and the entry SHALL NOT be created.
- The Access Control Cluster MAY fail to write, and return a RESOURCE_EXHAUSTED error, if an attempt is made to create or update an entry or extension such that storage is exhausted.

### ACCESS_RESTRICTED
- Attempts to access data model elements described by an entry in the ARL attribute for the accessing fabric SHALL result in an error of ACCESS_RESTRICTED.
- Attempts to access data model elements described by an entry in the CommissioningARL attribute during commissioning SHALL result in an error of ACCESS_RESTRICTED.

### ARL Review Outcomes
- If the device grants the request, the ARL attribute in the Access Control Cluster SHALL be updated to reflect the new access rights and a successful response SHALL be returned to the device making the request using the MTaer field of the callbackUrl.
- If the request is denied, the ARL attribute SHALL remain unchanged and a failure response SHALL be returned to the device making the request using the MTaer field of the callbackUrl.
